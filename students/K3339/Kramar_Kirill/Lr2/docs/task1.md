# Задача 1: сумма чисел 1..10^13

## Что требовалось

Нужно сравнить три подхода (`threading`, `multiprocessing`, `asyncio`) на задаче вычисления:

\[
1 + 2 + ... + 10\,000\,000\,000\,000
\]

## Почему нельзя честно итерировать до 10^13

Если делать прямой цикл на `10^13` шагов, программа будет работать очень долго.
Для учебной задачи важнее сравнить архитектурные подходы, а не «зависнуть» на одном цикле.

Поэтому используется математическая формула суммы арифметической прогрессии:

\[
S = \frac{(a_1 + a_n) \cdot n}{2}
\]

Идея такая:

- весь диапазон разбивается на чанки;
- каждый чанк считается отдельно;
- частичные суммы объединяются.

## Что именно сделано

Общие функции вынесены в `lr2.task1.common`:

- `split_range_inclusive(...)` — разбивает диапазон на равные части;
- `calculate_sum_for_range(...)` — сумма на диапазоне;
- `expected_sum_to_n(...)` — проверка результата.

### Вариант 1: `threading`

- файл: `lr2.task1.threading_sum`;
- создаются потоки, каждый считает свой чанк;
- результаты собираются в общий список.

### Вариант 2: `multiprocessing`

- файл: `lr2.task1.multiprocessing_sum`;
- создаётся `Pool`, каждому процессу отдаётся свой чанк;
- суммы возвращаются через `pool.map(...)`.

### Вариант 3: `asyncio`

- файл: `lr2.task1.async_sum`;
- создаются async-задачи;
- внутри используется `asyncio.to_thread(...)`, чтобы вынести синхронную функцию и показать async-оркестрацию через `await`.

## Результаты замера

Команда:

```bash
PYTHONPATH=src python -m lr2.task1.run_benchmark --workers 4
```

Результат:

| Подход | Время, сек |
|---|---:|
| threading | 0.000264 |
| multiprocessing | 0.103338 |
| asyncio | 0.003047 |

## Как интерпретировать

- Здесь вычисление в каждом чанке очень «короткое» (формула в O(1)), поэтому накладные расходы на создание процессов/тасков доминируют.
- `multiprocessing` проигрывает из-за высокой стоимости старта процессов и IPC.
- В реальной CPU-heavy задаче (долгие вычисления на каждом чанке) картина может быть другой, и процессы часто выигрывают у потоков из-за GIL.
