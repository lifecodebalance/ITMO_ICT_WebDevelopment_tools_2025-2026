# Сравнение и выводы

## Таблица по задаче 1 (сумма)

| Подход | Время, сек |
|---|---:|
| threading | 0.000264 |
| multiprocessing | 0.103338 |
| asyncio | 0.003047 |

Комментарий:

- На «короткой» работе внутри чанка быстрее всего оказалось то, что имеет минимум накладных расходов.
- Процессы в таком сценарии невыгодны из-за высокой цены запуска.

## Таблица по задаче 2 (парсинг)

| Подход | Время, сек |
|---|---:|
| threading | 0.940034 |
| multiprocessing | 1.278321 |
| asyncio | 0.896541 |

Комментарий:

- На I/O-bound задаче ожидаемо лучшие результаты у `threading` и `asyncio`.
- `asyncio` даёт масштабируемую архитектуру на большое число сетевых операций.
- `multiprocessing` имеет смысл чаще для тяжёлой CPU-обработки, а не для сетевого ожидания.

## Практические выводы

1. Нет «лучшего» подхода на все случаи.
2. Для CPU-bound в Python часто выбирают процессы (`multiprocessing`).
3. Для I/O-bound обычно лучше `asyncio` или потоки.
4. На маленьких задачах накладные расходы легко «съедают» выигрыш параллелизма.
5. При работе с вебом и БД очень важно:
   - ставить таймауты;
   - обрабатывать ошибки сети;
   - проверять SSL/сертификаты;
   - не блокировать event loop синхронными вызовами.
